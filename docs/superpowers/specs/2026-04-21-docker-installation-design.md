# Docker Installation — Design Spec

**Date:** 2026-04-21
**Scope:** Production-ready Docker installation for piighost, covering both
solo-workstation and small-cabinet-server deployments with a secure-by-default
posture and optional overlays for local embedder / local LLM.
**Status:** Awaiting implementation plan.

---

## Goal

Ship piighost as a turnkey Docker stack that a European legal/medical
professional can bring up with a single command, with strong default security,
encrypted backups, and a safe update path. Users who need local embeddings or
a local LLM layer can opt in via compose overlays without touching the base
stack.

## Non-goals

- Kubernetes manifests / Helm chart
- Multi-tenant SaaS deployment mode
- External secret-manager integrations (documented as patterns, not built-in)
- Hot config reload without restart

---

## Design decisions (brainstorming outcomes)

| # | Axis | Decision |
|---|---|---|
| 1 | Target deployment | Both: `workstation` + `server` profiles in one compose file |
| 2 | Stack scope | Core (MCP + daemon + vault) + opt-in overlays for embedder and local LLM |
| 3 | Image footprint | Two published tags: `slim` (~500 MB) + `full` (~3.5 GB) on GHCR |
| 4 | Security baseline | Forte par défaut: distroless base, Docker secrets, `cap_drop: ALL`, cosign signing, SBOM |
| 5 | Server auth | Bearer token by default; `PIIGHOST_AUTH=mtls` flips to mTLS |
| 6 | Persistence | Single `piighost-data` volume + age-encrypted nightly backup sidecar (7 daily + 4 weekly rolling) |
| 7 | Updates | Digest-pinned compose + `piighost self-update` CLI + passive update-notify sidecar |

## Architecture

### Container topology

```
Host ─ Claude Desktop / LangChain / Haystack client
         │
         ▼ HTTP loopback (workstation) | HTTPS via Caddy (server)
   ┌───────────────────────┐
   │ caddy  (server only)  │ ← TLS auto-LetsEncrypt
   └──────┬────────────────┘
          │
   ┌──────▼────────┐     ┌─────────────────────┐
   │ piighost-mcp  │ ──▶ │ piighost-daemon     │
   │ (FastMCP)     │     │ (anonymization core)│
   └───────────────┘     └──────────┬──────────┘
                                    │
                    ┌───────────────┼──────────────┐
                    ▼               ▼              ▼
       volume: piighost-data   (overlays)    sidecars
       ├── vault/              - embedder    ├── piighost-backup
       ├── projects/           - ollama      └── piighost-update-notify
       ├── index/
       └── audit/
```

### Networks

| Network | `internal` | Services attached |
|---|---|---|
| `piighost-internal` | yes | mcp, daemon, backup, notify |
| `piighost-public` | no | caddy (server profile only) |
| `piighost-llm` | yes | mcp, ollama (when overlay present) |

The `daemon` container has **no egress**. MCP gets egress only via
`piighost-llm` when the LLM overlay is active.

### Images

| Tag | Base image | Size target | Contents |
|---|---|---|---|
| `ghcr.io/jamon8888/hacienda-ghost:slim` | `gcr.io/distroless/python3-debian12:nonroot` | ~500 MB | piighost + regex detector + vault + MCP + daemon + CLI |
| `ghcr.io/jamon8888/hacienda-ghost:full` | `cgr.dev/chainguard/python:latest-dev` | ~3.5 GB | slim + GLiNER + sentence-transformers + LanceDB + haystack + langchain + kreuzberg |

**Image supply chain:**
- Published via GitHub Actions on push-to-main
- Signed with **cosign** (keyless OIDC)
- SBOM generated by **syft**, attached as OCI artifact
- Multi-arch: `linux/amd64` + `linux/arm64`
- Pinned by digest in `docker-compose.yml`, never by mutable tag

## Compose profiles & overlays

### Profiles inside `docker-compose.yml`

| Profile | Purpose | MCP transport / bind | Caddy | Auth | Backup |
|---|---|---|---|---|---|
| `workstation` (default) | Solo professional laptop | streamable HTTP on `127.0.0.1:8765` (loopback only) | — | off | on |
| `server` | Cabinet / on-prem VM | streamable HTTP behind Caddy TLS | yes | bearer (default) / mTLS (optional) | on |
| `no-backup` | Users with external backup | — | — | — | off |

MCP is served over streamable HTTP (FastMCP native transport), not stdio,
because Claude Desktop configures persistent servers by URL — not by
piping stdio into a short-lived container. Users set their Claude Desktop
`mcpServers` entry to `http://127.0.0.1:8765` (workstation) or
`https://piighost.cabinet.example/mcp` (server).

### Opt-in overlays

- **`docker-compose.embedder.yml`** — sentence-transformers HTTP service on `piighost-internal`. Eliminates egress to Mistral for embeddings.
- **`docker-compose.llm.yml`** — Ollama container on `piighost-llm`. Eliminates egress to any external LLM for inference.

### Canonical invocations

```bash
# Solo workstation
docker compose --profile workstation up -d

# Full sovereign cabinet stack
docker compose --profile server \
  -f docker-compose.yml \
  -f docker-compose.embedder.yml \
  -f docker-compose.llm.yml \
  up -d
```

## Security posture

### Container hardening (applied to every service)

```yaml
user: "10001:10001"
read_only: true
tmpfs: [/tmp, /run]
cap_drop: [ALL]
security_opt:
  - no-new-privileges:true
  - seccomp=default
pids_limit: 256
restart: unless-stopped
healthcheck: { test: [...], interval: 30s, timeout: 5s, retries: 3 }
```

### Secrets

All secrets delivered via Docker secrets (mounted at `/run/secrets/…`), never
as environment variables.

| Secret | Profile | Source file |
|---|---|---|
| `piighost_vault_key` | all | `docker/secrets/vault-key.txt` (chmod 600) |
| `piighost_bearer_tokens` | server (default) | `docker/secrets/bearer-tokens.txt` (one sha256 per line) |
| `piighost_mtls_ca_cert` | server + `PIIGHOST_AUTH=mtls` | `docker/secrets/ca.pem` |
| `piighost_mtls_ca_key` | server + `PIIGHOST_AUTH=mtls` | `docker/secrets/ca.key` |
| `piighost_age_recipient` | backup | `docker/secrets/age-recipient.txt` |

### Network isolation

- `piighost-daemon` has `internal: true` → **zero egress**
- `piighost-mcp` has egress only when attached to `piighost-llm` (overlay only)
- `caddy` is the only container with a published port on the server profile

## Persistence & backup

### Volumes

| Volume | Mount point | Purpose | Backed up |
|---|---|---|---|
| `piighost-data` | `/var/lib/piighost/` | vault, projects, index, audit | yes |
| `piighost-caddy` | `/data` | Caddy certs + state | yes |

### Backup sidecar (`piighost-backup`)

- Image: `piighost:slim` with `age` and `tar` in the base layer
- Schedule: 02:30 local time, cron-style
- Output: `./backups/piighost-YYYY-MM-DD.tar.age` (bind-mounted so users see files)
- Retention: 7 daily + 4 weekly, older snapshots auto-pruned
- Encryption: `age` with recipient key from Docker secret
- Opt-out: `docker compose --profile no-backup …`

### Restore flow

```bash
make restore BACKUP=./backups/piighost-2026-04-20.tar.age
```

Script halts the stack, replaces the `piighost-data` volume contents from the
decrypted tarball, and restarts.

## Update flow

### Digest pinning

`docker-compose.yml` pins every image by sha256 digest:

```yaml
image: ghcr.io/jamon8888/hacienda-ghost@sha256:abc123...
```

This prevents silent tag drift and makes rollback a one-line git revert.

### `piighost self-update` (new CLI subcommand)

1. Query GHCR API for latest digests of `slim` + `full` tags
2. Verify cosign signatures (abort on failure)
3. Show a diff: current digest → new digest + release notes
4. Prompt confirmation
5. Rewrite `docker-compose.yml` with new digests
6. `docker compose pull && docker compose up -d`
7. Emit exit code 0 on success; user can `git revert` the compose file to roll back

### Passive update-notify sidecar (`piighost-update-notify`)

- Polls GHCR every 24h
- Writes a human-readable line to stderr: `[piighost] update available: slim abc123… → def456…`
- Writes machine-readable `/var/lib/piighost/update-available.json`
- **Never** modifies the running stack

## Acceptance criteria

A working implementation must satisfy all of:

- [ ] `docker compose --profile workstation up -d` brings up MCP + daemon + backup + notify from prebuilt images, Claude Desktop can talk to the MCP container via stdio/loopback
- [ ] `docker compose --profile server …` brings up the server stack with Caddy TLS, bearer-token auth enforced, mTLS available via `PIIGHOST_AUTH=mtls`
- [ ] Both images build cleanly in CI, publish to GHCR, carry cosign signatures + SBOM
- [ ] `make backup` produces a dated `.tar.age` file; `make restore BACKUP=…` round-trips the vault to a fresh volume
- [ ] `piighost self-update` rewrites compose digests atomically and rolls back on signature failure
- [ ] Update-notify sidecar emits a stderr line and JSON file when a newer digest exists
- [ ] All containers run as UID 10001, read-only root FS, `cap_drop: ALL`
- [ ] No secret is visible in `docker inspect` output
- [ ] README.fr.md has a complete `## Installation Docker` section in French only
- [ ] Existing 774-test suite still passes (no regressions)

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Distroless base lacks shell → hard to debug | Provide `cgr.dev/chainguard/python:latest-dev` for `full`; document `docker run --rm -it --entrypoint=python … ` pattern for `slim` |
| Cosign verification adds CI latency | Cache OIDC tokens; verification is <2s |
| Backup sidecar holds the same volume as daemon → lock contention | Use `sqlite3` online backup API for vault; `tar` tolerates eventual-consistency on index |
| Users skip backup setup → data loss | Backup enabled by default; only `--profile no-backup` disables it |
| Bearer token leakage via shell history | `piighost docker init` creates the file directly; README warns against `echo … >` |
| Claude Desktop doesn't support mTLS today | Bearer is the default; mTLS is explicit opt-in for users who need it |
| Image size grows silently | CI job fails if `full` > 4 GB or `slim` > 700 MB |
